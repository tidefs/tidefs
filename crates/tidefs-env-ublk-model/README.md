# tidefs-env-ublk-model

Bounded pure model for the uBLK qid/tag request lifecycle behind
`ublk.qid_tag.exactly_once_completion.v1`.

The crate models queue/tag allocation, in-flight ownership, completion, abort,
timeout, recovery, and reissue transitions. Legal uBLK read, write, flush, and
discard events refine into the current TideFS block request/completion contract
types in `tidefs-types-vfs-core`.

This is model evidence only. It does not run a uBLK daemon, issue ioctls,
submit block I/O, or claim full block-volume runtime correctness.

## Started-export service-loop receipt fixture

`ublk.started_export.live_service_loop.v1` remains a planned claim behind a
claims-gate review path. The `started_export` module extends the pure uBLK
environment model with a deterministic started-export service-loop receipt
fixture.

The fixture records:
- qid/tag generation across configured queues and tags
- queue bounds relative to `UBLK_MAX_NR_QUEUES` and `UBLK_MAX_QUEUE_DEPTH`
- completion outcomes (complete, abort, timeout, reissue-after-timeout)
- exact unsupported opcode handling for every uBLK opcode outside the current
  contract set (write_same, write_zeroes, zone_*, report_zones)
- which request classes were exercised (read, write, flush, discard)

The receipt is generated by `generate_started_export_receipt(queue_count,
queue_depth, seed)` and is fully deterministic for identical inputs. Tests
verify determinism and reject queue count/depth values outside the uBLK ABI
bounds.

### Evidence tier

This fixture is **model-tier evidence** only. It does not satisfy
`runtime-ublk-started-export-admission-artifact`. No uBLK daemon, qemu, fio,
mkfs, or mount behavior is exercised or claimed. The fixture demonstrates the
model's coverage of the service-loop lifecycle but does not prove live daemon
correctness.

See the `started_export` module documentation and the generated fixture at
`validation/artifacts/ublk/started-export-service-loop-model.json` for the
deterministic receipt output.
