== tidefsctl

`tidefsctl` is the TideFS command-line interface. Its job is to expose the
operator path without pretending that every bring-up tool is final product
surface.

=== Starting Points

Most operator sessions should start with one of these groups:

* `tidefsctl pool --help` for pool creation, discovery, import/export, status,
  and properties;
* `tidefsctl dataset --help` for the pool-wide dataset catalog and dataset
  properties;
* `tidefsctl snapshot --help` for snapshot lifecycle and transfer commands;
* `tidefsctl block --help` for ublk block-volume attachment and inspection;
* `tidefsctl diag --help` for redacted support bundles.

Development sessions may use `tidefsctl mount`, `tidefsctl pool mount`, and
the `cluster * exercise` commands. Those are harnesses, not the final kernel
runtime.

=== Pool Identity And Routing

The pool name should be the normal positional identity for operator commands,
similar to established filesystem CLIs. The name is not a shorthand for
opening some local directory behind the runtime.

When a pool is imported or mounted, its live state is cached by and belongs to
the active runtime owner:

* in final kernel mode, `tidefsctl` must talk to the declared kernel UAPI;
* in userspace mode, `tidefsctl` must talk to the FUSE or ublk owner that
  imported the pool and holds its caches, admission state, and runtime view.

That owner interface is not optional plumbing. A pool-name-only command must
not reopen storage behind the owner and then guess at state. If the kernel or
daemon owner does not implement a requested operation yet, the command must
fail closed instead of falling back to direct device access.

The current userspace FUSE owner exposes an imported-pool JSON socket under
`/run/tidefs/pools/<pool-uuid>/owner.sock`, with a manifest at
`owner.json`. Owner discovery may use the pool UUID or a pool-name registry
entry, but the reachable socket is the interface. A manifest alone is only
runtime residue and is not authorization to reopen devices directly. That
socket currently handles live `pool status`, pool properties, dataset catalog
create/list/rename/destroy and dataset property
queries/mutations, dataset set-strategy/upgrade feature mutations, snapshot
create/list/destroy/rollback, full `snapshot send --output` streams, dataset
key sealing/rotation, and owner-mediated `pool export` shutdown. Those
operations run against the mounted engine that owns cached state. Snapshot
receive, live destroy, live network/incremental snapshot send, and live block
attach/export/send/receive still fail closed where the owner-side
implementation is missing. Live device removal is implemented for the current
userspace FUSE owner by calling the mounted pool's evacuation path through
that owner; it refuses forced removal and still needs follow-up active-label
topology persistence before it can close the full device lifecycle debt.

Offline inputs are explicit. Commands that operate on a pool that is not
imported must say how to find it with arguments such as `--devices` or
`--backing-dir`. Those arguments authorize offline work on exported storage,
repair, discovery, or an owner-creating import-and-mount harness. Dataset,
snapshot, and property commands must not turn an exported device set into an
imported runtime as a side effect. Those commands first read the storage
identity, and if the live-owner registry already names that pool UUID, the
command has to route through the owner of that exact live state instead of
opening the devices directly. An `ACTIVE` label on disk is
cached recovery evidence rather than the live interface itself, but it still
means the CLI must not treat the devices as ordinary exported storage. With no
reachable owner, live-state commands fail closed; recover or create the owner
with `pool mount <pool> <mountpoint> --devices ...`, then operate through that
owner.

The current userspace path that both imports and creates a live owner is
`pool mount <pool> <mountpoint> --devices ...`. `pool import <pool>` is an
owner-mediated request for an imported pool. `pool import <pool> --devices ...`
does not activate an exported pool by itself in the current CLI because a
one-shot device import would create live state and then leave no owner for
cached state. A future kernel import path must publish a kernel owner
interface before `pool import <pool> --devices ...` can activate storage.
After a pool is live, follow-up commands must use the owner interface or fail
closed.

Do not implement positional pool-name usability by reopening
`/run/tidefs/pools/<name>`, by treating the pool name as a backing path, or by
calling storage libraries behind the live owner. That would bypass cached
state and split authority. Add the owner client first, then make the positional
form call it.

Common owner-aware forms put the pool first:

[source,sh]
----
tidefsctl pool status <pool>
tidefsctl pool import <pool>
tidefsctl pool get <pool> <property>
tidefsctl pool set <pool> <property>=<value>
tidefsctl dataset create <pool> <dataset>
tidefsctl dataset list <pool>
tidefsctl dataset set <pool> <dataset> <property>=<value>
tidefsctl dataset seal-key <pool> <dataset> --passphrase <phrase>
tidefsctl dataset rotate-key <pool> --old-passphrase <phrase> --old-salt <hex> --new-passphrase <phrase>
tidefsctl snapshot create <pool> <snapshot>
tidefsctl snapshot list <pool>
tidefsctl pool export <pool>
tidefsctl pool destroy <pool>
tidefsctl device remove <pool> <device>
tidefsctl block attach <pool>
tidefsctl block send <pool> --target-addr <addr>
tidefsctl block receive <pool> --source-addr <addr>
----

The exported/offline device-removal form spells its storage handles
explicitly:

[source,sh]
----
tidefsctl device remove <pool> <device> --backing-dir <target-store> --surviving-dirs <stores>
tidefsctl pool import <pool> --devices <devices>
tidefsctl pool destroy <pool> --devices <devices>
tidefsctl block attach <pool> --backing-dir <object-store>
tidefsctl block send <pool> --backing-dir <object-store> --target-addr <addr>
tidefsctl block receive <pool> --backing-dir <object-store> --source-addr <addr>
----

The device-removal offline form first probes existing labels without creating
or opening the store writable. Those labels provide topology and recovery
evidence. If the labels identify `ACTIVE` imported state, the request routes to
the owner interface or fails closed; it does not continue into direct
evacuation from cached live state. Object-store `--backing-dir` forms are
explicit development paths. If a reachable owner manifest names that backing
directory, snapshot commands route to that owner; block commands route only
when the manifest also names the requested pool. If an owner manifest names
the backing directory for a different pool, the command refuses instead of
opening the cached imported state as exported storage. These forms must not
route by pool name alone or make pool-name-only live state changes behind the
owner.
Device removal survivor directories and device rebuild source/replacement
directories are offline object-store paths too, and must be refused when they
point into `/run/tidefs/pools` or are named by an imported-pool owner manifest.
`pool integrity-check <path>` follows the same rule: a path owned by an
imported pool routes to that owner or fails closed, while direct scans are only
for exported/offline storage.

The old `--pool <pool>` spelling is not the operator form. It must not be
kept as a parallel control path for imported state.

=== Command Classes

[cols="1,2",options="header"]
|===
| Class | Commands

| Current operator surface
| `pool create`, `pool scan`, `pool status`, owner-mediated `pool import`,
  `pool export`, `pool destroy`, `pool integrity-check`, `pool get`, `pool set`,
  `pool list-props`; `dataset create`, `dataset list`, `dataset destroy`,
  `dataset rename`, `dataset set-strategy`, `dataset seal-key`,
  `dataset rotate-key`, `dataset upgrade`, `dataset get`, `dataset set`,
  `dataset list-props`; `snapshot create`, `snapshot list`, `snapshot destroy`,
  `snapshot rollback`, `snapshot send`, `snapshot receive`; `device remove`,
  `device rebuild`; `block attach`, `block detach`, `block list`,
  `block send`, `block receive`; `defrag`; `diag`

| Development harness
| `mount`, `pool mount`, hidden `pool create --file-devices`,
  `cluster placement exercise`, `cluster heal exercise`

| Cluster operator prototype
| `cluster pool create`
|===

The command classes are part of the CLI contract. Help text, parser tests, and
book text should change together when a command moves between classes.

=== Output Rules

Operator commands should prefer plain text that is easy to scan and provide
`--json` for automation when the command has stable structured output.

Errors should name the command path first, for example
`tidefsctl pool import: ...`, then describe the failed operation and the
resource involved. Warnings must not hide pre-alpha limitations behind success
language.

=== Help Text Rule

Top-level help should answer two questions quickly:

* Is this command an operator surface or a development harness?
* Which subcommand should the operator inspect next?

Keep command descriptions short, concrete, and honest. Avoid resurrecting
removed control-plane stubs or hiding a development harness behind final
operator wording.

=== Final Kernel Path

The final full-kernel direction leaves `tidefsctl` as an operator/query client.
It may request configuration changes, inspect state, trigger runbook
operations, and collect support bundles. It must not become a required
userspace daemon for normal mounted filesystem operation.
