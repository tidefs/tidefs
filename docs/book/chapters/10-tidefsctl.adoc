== tidefsctl

`tidefsctl` is the TideFS command-line interface. Its job is to expose the
operator path without pretending that every bring-up tool is final product
surface.

The operator UAPI authority decision is recorded in
`docs/OPERATOR_UAPI_AUTHORITY.md`. This chapter consumes the checked
command-classification and admission table that decision references; it does
not turn harnesses, passive kernel probes, ublk/FUSE preview hooks, or cluster
prototypes into a production ABI, a kernel-module ABI freeze,
kernelspace-readiness evidence, or final distributed operator UAPI.

=== Starting Points

Most operator sessions should start with one of these groups:

* `tidefsctl pool --help` for pool creation, discovery, import/export, status,
  and properties;
* `tidefsctl dataset --help` for the pool-wide dataset catalog and dataset
  properties;
* `tidefsctl snapshot --help` for snapshot lifecycle and transfer commands;
* `tidefsctl storage-intent --help` for read-only supplied policy, receipt,
  and evidence-query explanation;
* `tidefsctl block --help` for ublk block-volume attachment and inspection;
* `tidefsctl kernel --help` for the declared kernel control endpoint probe;
* `tidefsctl diag --help` for redacted support bundles.

Development sessions may use `tidefsctl mount`, `tidefsctl pool mount`, and
the `cluster * exercise` commands. Those are harnesses, not the final kernel
runtime.

=== Pool Identity And Routing

The pool name should be the normal positional identity for operator commands,
similar to established filesystem CLIs. The name is not a shorthand for
opening some local directory behind the runtime.

When a pool is imported or mounted, its live state is cached by and belongs to
the active runtime owner:

* in final kernel mode, `tidefsctl` must talk to the declared kernel UAPI;
* in userspace mode, `tidefsctl` must talk to the FUSE or ublk owner that
  imported the pool and holds its caches, admission state, and runtime view.

That owner interface is not optional plumbing. A pool-name-only command must
not reopen storage behind the owner and then guess at state. If the kernel or
daemon owner does not implement a requested operation yet, the command must
fail closed instead of falling back to direct device access.

The current userspace FUSE owner exposes an imported-pool JSON socket under
`/run/tidefs/pools/<pool-uuid>/owner.sock`, with a manifest at
`owner.json`. Owner discovery may use the pool UUID or a pool-name registry
entry, but the reachable socket is the userspace interface. In kernel mode the
interface is the declared kernel UAPI client, not the cached manifest. A
manifest alone is only
runtime residue and is not authorization to reopen devices directly. That
socket currently handles live `pool status`, pool properties, dataset catalog
create/list/rename/destroy and dataset property
queries/mutations, dataset set-strategy/upgrade feature mutations, snapshot
create/list/destroy/rollback, full `snapshot send --output` streams, dataset
key sealing/rotation, and owner-mediated `pool export` shutdown. Those
operations run against the mounted engine that owns cached state. Snapshot
receive, live destroy, live network/incremental snapshot send, and live block
attach/export/send/receive still fail closed where the owner-side
implementation is missing. A follow-up `pool mount <pool> <mountpoint>` for an
already-imported userspace FUSE pool also fails closed until the owner can
create an additional mount/session itself; returning success without mounting
would be a false live-state operation. Live device removal is implemented for
the current userspace FUSE owner by calling the mounted pool's evacuation path
through that owner; it refuses forced removal and still needs follow-up
active-label topology persistence before it can close the full device lifecycle
debt.

Offline inputs are explicit. Commands that operate on a pool that is not
imported must say how to find it with byte-addressable `--devices` arguments
where the command supports offline access. Those arguments authorize offline
work on exported storage, repair, discovery, or an owner-creating
import-and-mount harness. Dataset, snapshot, and property commands must not
turn an exported device set into an imported runtime as a side effect. Those
commands first read the storage identity, and if the live-owner registry
already names that pool UUID, the command has to route through the owner of
that exact live state instead of opening the devices directly. An `ACTIVE`
label on disk is cached recovery evidence rather than the live interface
itself, but it still means the CLI must not treat the devices as ordinary
exported storage. With no reachable owner, live-state commands fail closed;
recover or create the owner with `pool mount <pool> <mountpoint> --devices
...`, then operate through that owner.

The current userspace path that both imports and creates a live owner is
`pool mount <pool> <mountpoint> --devices ...`. `pool import <pool>` is an
owner-mediated request for an imported pool. `pool import <pool> --devices ...`
does not activate an exported pool by itself in the current CLI because a
one-shot device import would create live state and then leave no owner for
cached state. A future kernel import path must publish a kernel owner
interface before `pool import <pool> --devices ...` can activate storage.
After a pool is live, follow-up commands must use the owner interface or fail
closed.

Do not implement positional pool-name usability by reopening
`/run/tidefs/pools/<name>`, by treating the pool name as a backing path, or by
calling storage libraries behind the live owner. That would bypass cached
state and split authority. Add the owner client first, then make the positional
form call it.

Common owner-aware forms put the pool first:

[source,sh]
----
tidefsctl pool status <pool>
tidefsctl pool import <pool>
tidefsctl pool integrity-check <pool>
tidefsctl pool get <pool> <property>
tidefsctl pool set <pool> <property>=<value>
tidefsctl dataset create <pool> <dataset>
tidefsctl dataset list <pool>
tidefsctl dataset set <pool> <dataset> <property>=<value>
tidefsctl dataset seal-key <pool> <dataset> --passphrase <phrase>
tidefsctl dataset rotate-key <pool> --old-passphrase <phrase> --old-salt <hex> --new-passphrase <phrase>
tidefsctl snapshot create <pool> <snapshot>
tidefsctl snapshot list <pool>
tidefsctl pool export <pool>
tidefsctl pool destroy <pool>
tidefsctl device remove <pool> <device>
tidefsctl block attach <pool>
tidefsctl block send <pool> --target-addr <addr>
tidefsctl block receive <pool> --source-addr <addr>
----

Exported/offline byte-addressable forms spell their devices explicitly:

[source,sh]
----
tidefsctl pool import <pool> --devices <devices>
tidefsctl pool destroy <pool> --devices <devices>
tidefsctl pool integrity-check <pool> --devices <devices>
----

Pool members are named with `--devices` and must be block devices, or regular
files only in hidden development mode. Directory object-store roots may remain
internal compatibility and harness storage, but they are not operator pool
media, block-volume backing, snapshot storage, or device lifecycle handles.
Device removal, rebuild, repair, and block-volume operations must consume pool
labels, byte-addressable media, placement receipts, and the live owner instead
of receiving directory survivor/replacement paths from the operator.
`pool integrity-check <pool>` follows the same rule as the other pool-name
commands: it routes to the owner interface for imported state. Direct scans are
only for exported/offline or not-yet-imported storage named explicitly with
`--devices`; if those inputs identify imported state, the command routes to the
exact owner or fails closed.

The old `--pool <pool>` spelling is not the operator form. It must not be
kept as a parallel control path for imported state.

=== Storage Intent Explanation

`tidefsctl storage-intent explain --input <json-bundle>` is a read-only,
passive-diagnostic rendering surface. It consumes supplied
`tidefs-storage-intent-core` records (policy, receipts, evidence-query
snapshots, refusal records, data-shape, layout, relocation, cost/wear, and
prefetch/residency decisions) and produces a source-qualified, evidence-cited
operator explanation. It deliberately does not scan topology, reopen pool
state, or infer missing producer evidence.

==== Basic Usage

....
tidefsctl storage-intent explain --input evidence.json
tidefsctl storage-intent explain --input evidence.json --dataset mypool/mydataset
tidefsctl storage-intent explain --input evidence.json --file /data/example --range 0..4096
tidefsctl storage-intent explain --input evidence.json --json | jq .
....

`--dataset`, `--file`, and `--range` supply subject labels when the input
bundle does not carry its own subject declaration. `--json` emits a
machine-parseable JSON report with sections, availability levels, and
enumerated facts.

==== Input Format

The input is a single JSON object with optional fields:

[source,json]
....
{
  "subject": { "dataset": "mypool/myds", "file": "/data/example", "range": "0..4096" },
  "policy": { /* StorageIntentPolicy */ },
  "receipts": [ /* StorageIntentReceipt[], empty when none */ ],
  "query_snapshot": { /* StorageIntentEvidenceQuerySnapshot */ },
  "refusals": [ /* StorageIntentRefusal[], empty when none */ ],
  "read_freshness": { /* ReadSourceFreshnessRecord */ },
  "data_shape": { /* DataShapeRecord */ },
  "layout": { /* LayoutAllocatorRecord */ },
  "relocation": { /* RelocationLifecycleRecord */ },
  "cost_wear": { /* CostWearRecord */ },
  "prefetch_residency": { /* PrefetchResidencyDecisionRecord */ }
}
....

Every field is optional and defaults to absent. When a field is absent, the
corresponding output section renders as `unavailable` with a source-qualified
note naming the expected producer or evidence family. The renderer must not
infer placement, durability, or read-serving facts from local caches, device
probes, or benchmarks.

==== Output Format (Plain Text)

The plain-text output renders one section per evidence family. Each section is
prefixed with an availability tag (`available`, `limited`, `unavailable`,
`degraded-visible`, `blocked`, `refused`, or `unsafe-visible`) and a source
annotation:

....
storage-intent explanation
  input-source: /path/to/evidence.json
  subject: dataset=mypool/myds file=/data/example range=0..4096

evidence-query-snapshot [available; source: #913 supplied StorageIntentEvidenceQuerySnapshot]
  - snapshot-id: a1b2c3d4...
  ...

requested-policy [available; source: supplied StorageIntentPolicy]
  - policy-id: ...
  - requested-guarantee: local-intent
  ...

earned-acknowledgment [available; source: supplied StorageIntentReceipt set]
  - receipt ... earned ack durable-intent-zero-lag ...
  ...
....

When the input carries a `query_snapshot`, the renderer cites the #913
snapshot identity, included evidence refs, per-family freshness state, and
completeness verdict. Missing, stale, or refused snapshot families render their
individual state. When no snapshot is supplied, the section renders
`unavailable` with a note that multi-family explanation is non-authority and
must not be replaced by topology scans.

==== Evidence Families Rendered

The renderer sections, in order:

- `evidence-query-snapshot` -- #913 snapshot cut identity, completeness, per-family freshness
- `requested-policy` -- compiled policy identity, guarantee class, durability/media/trust floor, workload prediction
- `earned-acknowledgment` -- per-receipt ack class, policy revision, durability state, observed lag
- `placement-receipts` -- per-receipt placement media/role/read-source and authority class
- `read-serving` -- selected read source, freshness frontier, geo lag, prefetch/residency decision
- `remote-rpo-lag` -- remote/geo receipt lag, read-source lag, durability-lag
- `trust-domain` -- per-receipt trust flags, session security, key epoch, residency, sharing domain, compromise/quarantine
- `volatility-loss` -- volatile receipt role/media explicit loss table
- `pending-work` -- relocation/rebake/repair lifecycle state and skipped reasons
- `data-shape` -- selected transform/EC/archive shape, compression/dedup/encryption state
- `layout-allocator` -- fragmentation, free-run pressure, alignment, zone/write-pointer state
- `prediction` -- shadow/trial/admitted move state, cooldown, payback, skipped reasons
- `cache-authority` -- cache-only vs authoritative RAM/flash/durable placement class
- `cost-wear-reserve` -- movement debt, write amplification, payback window, critical reserves
- `refusals` -- explicit refusal records with reasons for each refused guarantee upgrade

==== Non-Authority And Claims-Gate Boundary

The explain surface is a read-only passive diagnostic. It does not execute
placement, change acknowledgments, measure transport, or retire receipts.
Output reflects receipt truth only; it does not make product, performance,
durability, RAM, WAN, or wear-protection claims. Help text and docs that frame
storage intent as a successor claim must be gated through #875 registered
claim ids and fail closed until validated. The explain renderer never hides
volatile or unsafe receipts from operator output or support bundles.

=== Storage Intent Policy Source

`tidefsctl storage-intent policy set|clear|show|dry-run` is the first
dataset-scoped prefetch/residency UAPI slice. It builds and renders an
explicit JSON policy source for adaptive prefetch, cache-only trials, RAM/PMem,
flash/HDD, WAN/object/archive staging, promotion, demotion, dwell, cooldown,
payback, feedback, and refusal controls. `set` and `clear` are local-only
source publication helpers; `show` and `dry-run` are passive renderers.

The policy source is not runtime authority. `dry-run` routes the staged source
through the #855 policy compiler, renders #901 rollout state as staged or
blocked when rollout/convergence evidence is absent, renders #926 preflight as
required-unavailable or not-covered, and renders #972 executor support for each
requested action as available, blocked, or refused. It does not execute
prefetch, promotion, demotion, data movement, source retirement, or budget
spend, and it does not reinterpret old receipts or already-running work.

[source,sh]
----
tidefsctl storage-intent policy set tank/fs \
  --allow bounded-readahead,cache-only-trial \
  --refuse flash-hot-serving \
  --max-prefetch-window-bytes 1048576 \
  --budget-owner dataset-a \
  --budget ram=dataset-a \
  --feedback-mode prefetch-windows \
  --output policy-source.json

tidefsctl storage-intent policy dry-run --input policy-source.json
tidefsctl storage-intent policy show --input policy-source.json --json
tidefsctl storage-intent policy clear tank/fs --all --output policy-clear.json
----

Pool defaults, mount profiles, caller hints, workload detection, and learned
feedback may seed or tighten the compiled source only through dataset
inheritance and #855 merge rules. They must not relax a stricter dataset
policy, authorize promotion by hotness hint alone, borrow another dataset's
budget, or hide missing wear/cost/RPO/trust/preflight evidence.

=== Kernel Runtime Inventory

`tidefsctl kernel status` reports the declared kernel control endpoint and the
source-visible TideFS kernel runtime surfaces without opening the endpoint or
issuing ioctls. The default control endpoint is `/dev/tidefs-control`, and
`--control-dev <path>` may point the probe at a different path for tests or
operator diagnostics.

The current command is an honesty boundary, not proof of full-kernel control.
It reports whether the path is missing, present as a character device, or
present with the wrong file type. It also reports that the current kernel
source does not wire TideFS-owned kthreads or runtime workqueues as product
surfaces. It always reports that production kernel UAPI wiring is not
implemented until a later kernel-control issue wires a real client. The passive
report names the minimum future control contract: the declared TideFS control
character-device identity, a versioned readonly handshake, version, status, and
capabilities status calls, mutating-operation refusal until admission and owner
proof exist, and the `pre-alpha-no-production-abi-freeze` compatibility boundary.
It must not open `/dev/tidefs-control`, issue control ioctls, or treat
`/run/tidefs/pools/.../owner.json` manifests as kernel authority.

[source,sh]
----
tidefsctl kernel status
tidefsctl kernel status --json
tidefsctl kernel status --control-dev /dev/null --json
----

The JSON form exposes stable passive-inventory fields including
`status_is_passive`, `control_endpoint_opened`,
`readonly_status_calls_issued`, `mutating_operation_admission`,
`owner_authority_proof`, `tidefs_owned_kthreads_wired`, and
`tidefs_owned_workqueues_wired`.

=== Output Rules

Operator commands should prefer plain text that is easy to scan and provide
`--json` for automation when the command has stable structured output.

Errors should name the command path first, for example
`tidefsctl pool import: ...`, then describe the failed operation and the
resource involved. Warnings must not hide pre-alpha limitations behind success
language.

=== Diagnostic Bundles

`tidefsctl diag` collects a redacted, read-only support bundle. The bundle is
operator evidence, not storage authority and not claim validation. Every major
section carries a source label such as `passive-host-probe`,
`command-classification-registry`, `offline-device-scan`, `live-owner`, or
`unavailable`.

Passive host probes cover system, build, and environment facts. The command
surface section is generated from the command-classification registry. Explicit
`--devices` input is an offline device scan source only: exported pool labels
may produce exported/offline pool facts, while `ACTIVE` imported-pool labels are
reported as label-only evidence that requires a live owner before deeper state
is read. Dataset and validation/claim-adjacent sections are marked
`unavailable` when no live-owner or validation-row source was consulted, rather
than emitting zero-filled pass/fail taxonomies.

`tidefsctl diag --json` prints the same source-qualified bundle to stdout. The
default file-output mode also names the report source and maturity before
printing the bundle path.

=== Help Text Rule

Top-level help should answer two questions quickly:

* Is this command an operator surface or a development harness?
* Which subcommand should the operator inspect next?

Keep command descriptions short, concrete, and honest. Avoid resurrecting
removed control-plane stubs or hiding a development harness behind final
operator wording.

=== Final Kernel Path

The final full-kernel direction leaves `tidefsctl` as an operator/query client.
It may request configuration changes, inspect state, trigger runbook
operations, and collect support bundles. It must not become a required
userspace daemon for normal mounted filesystem operation.
