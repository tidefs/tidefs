## 3. Phase-by-Phase Summary

### 3.1 Phase 5: ViewBuilderService

**Crate**: `tidefs-derived-catalog` (new). Maintains derived catalog views —
cached directory entries, sorted index views, and polymorphic directory index
structures. Wraps an `IncrementalJob` that walks the authoritative directory
B-tree, rebuilds stale derived views, serves in-memory cached entries, evicts
cold entries under memory pressure, and compacts fragmented index pages.

**Key data structures**: `ViewEntry` (dir handle, generation, last_access, dirty),
`ViewCache` (LRU eviction, capacity in bytes), `ViewBuilderJob` (IncrementalJob,
`JobKind::ViewBuilder`, single persistent cursor).

### 3.2 Phase 6: DataCleanerService

**Crates**: `tidefs-cleanup-queue-core` (existing), `tidefs-cleanup-job-core`
(existing). Processes refcount delta queues generated by the deferred cleanup
subsystem. Drains the per-family delta queue in budgeted batches, applies deltas
to the authoritative refcount B-tree, and handles underflow detection.

**Key data structures**: `DataCleanerJob` (IncrementalJob, `JobKind::DataCleaner`,
single `QueueFamily` cursor), `CleanupBatch` ≤ 256 entries per tick.

### 3.3 Phase 7: SegmentCleanerService

**Crates**: `tidefs-reclaim` (existing), `tidefs-reclaim-job-core` (existing).
Reclaims dead segments identified by the space accounting system. Evaluates
benefit/cost ratio before cleaning, relocates live extents, and frees the segment.

**Key data structures**: `SegmentCleanerJob` (IncrementalJob, `JobKind::SegmentCleaner`,
cursor on segment list), `CleanBenefit` threshold, `RelocationBatch`.

### 3.4 Phase 8: FUSE Integration

**Crates**: `tidefs-fuse` (existing). Wires the `BackgroundScheduler` into the
FUSE daemon main loop. After each FUSE request batch, the scheduler runs one
cycle with demand preemption. Under high foreground pressure, the scheduler
reduces background budget or skips cycles entirely.

**Key contract**: `plan_cycle(budget) -> CycleReport` called inline on the FUSE
event loop thread. Demand preemption with bounded deadline (500µs planning
overhead).

### 3.5 Phase 9: CompactionService

**Crates**: `tidefs-derived-catalog` (shared with P5). Compacts derived catalog
B-tree pages and refcount B-tree pages that exceed fragmentation thresholds.
Uses B-tree sibling-range locking to avoid conflicts with foreground operations.

**Key data structures**: `CompactionJob` (IncrementalJob, `JobKind::Compaction`,
cursor on B-tree node ranges), `FragmentThreshold` (≥ 50% dead entries).


**Crate**: `tidefs-xtask` (existing). Adds `check-background-service-framework`

- All 11 services register and tick without errors under synthetic workloads
- Budget enforcement: no service exceeds its per-tick budget
- Priority ordering: higher-priority services always dispatch before lower
- Round-robin fairness: within a priority stage, no service starves
- Deterministic replay: identical initial state + identical budgets → identical
  `TickLog` output

## 4. Key Data Structures

All phase 5–9 services conform to the Phases 1–4 contracts:

```rust
pub trait IncrementalJob: Send + Sync {
    fn tick(&mut self, budget: &WorkBudget) -> Result<StepResult, JobError>;
    fn checkpoint(&self) -> Result<Checkpoint, JobError>;
    fn resume(checkpoint: Checkpoint) -> Result<Self, JobError>;
    fn job_kind(&self) -> JobKind;
    fn priority(&self) -> ServicePriority;
}

pub struct WorkBudget {
    pub max_items: usize,   // max logical work units
    pub max_bytes: usize,   // max I/O bytes
    pub max_ms: u64,        // max wall-clock ms
}

pub enum StepResult {
    Advanced(usize),   // n items processed
    Exhausted,         // no more work
    Throttled,         // budget exhausted
    Error(JobError),   // non-fatal error
}
```

Each phase 5–9 service adds phase-specific types on top of these contracts.

## 5. Algorithms

### 5.1 View Builder (P5)

1. Load checkpoint cursor into authoritative directory B-tree
2. Scan forward: for each directory entry, check if derived view is stale
   (generation mismatch)
3. If stale: rebuild view from authoritative B-tree, write to view cache
4. If cache at capacity: evict LRU entry, compact if fragmentation ≥ 50%
5. Advance cursor, persist checkpoint
6. Emit `TickReport` with entries_built, entries_evicted, pages_compacted

### 5.2 Data Cleaner (P6)

1. Dequeue up to `budget.max_items` entries from reclaim queue by family
2. For each entry: apply refcount delta to authoritative B-tree
3. Check for underflow (refcount < 0): log integrity error, skip entry
4. Track aggregates: bytes_reclaimed, entries_processed, underflows_detected
5. Persist cursor as (queue_family, last_key)

### 5.3 Segment Cleaner (P7)

1. Scan segment list from cursor, compute benefit/cost ratio
2. If benefit/cost < threshold: skip segment
3. If ≥ threshold: read live extents, relocate to new segment, mark old segment free
4. Track bytes relocated, segments cleaned
5. Persist cursor as last segment id

### 5.4 FUSE Scheduler Loop (P8)

```
fuse_main_loop:
  while running:
    await FUSE request batch
    process_requests()
    if demand_pressure < PRESSURE_THRESHOLD:
      budget = compute_background_budget(demand_pressure)
      scheduler.plan_cycle(budget)
    else:
      skip_background_cycle()
```

### 5.5 Compaction (P9)

1. Scan B-tree node ranges from cursor
2. For each node with dead_entries ≥ 50% of capacity:
   - Lock node + left sibling + right sibling
   - Merge with left sibling if combined size ≤ node capacity
   - Otherwise rewrite node in-place dropping dead entries
3. Advance cursor, persist checkpoint


1. Build synthetic pool with all 11 services registered
2. Run 100 deterministic cycles with fixed seed
3. Assert: no `StepResult::Error` unless injected
4. Assert: budget enforcement (per-tick item/byte/ms caps)
5. Assert: priority ordering (Critical before LatencySensitive before …)
6. Assert: round-robin fairness (every service in stage gets a turn)
7. Assert: deterministic replay (capture `TickLog`, replay, compare)

## 6. Tradeoffs

| Decision | Rationale |
|----------|----------|
| **Data cleaner: inline vs. batch delta application** | Batch application (≤ 256 entries/tick) amortizes B-tree locking overhead while bounding per-tick latency to ~60ms |
| **Segment cleaner: eager vs. lazy cleaning** | Lazy with benefit/cost threshold. Eager cleaning wastes I/O on segments with few dead extents; lazy allows dead space to accumulate under high free-space headroom |
| **FUSE integration: inline vs. dedicated thread** | Inline on FUSE event loop. Planning overhead < 500µs for < 20 services. Dedicated thread adds synchronization complexity without measurable benefit below 20-service threshold |
| **Compaction: sibling-lock vs. full-tree lock** | Sibling-range locking (node ± siblings). Full-tree lock blocks all foreground B-tree ops; sibling locking limits blocking to adjacent key ranges |
